# Service and prediction problems

Дата разбора: 2026-06-09.

Источник текущего анализа:

- `data/interim/analysis/export_20260609_121758/`
- `forecast_vs_fact_joined_2026-05-23_2026-06-09.csv`
- `flight_status/kunashir_flight_status_hourly.csv`
- `forecast_monitor_exports/forecast_outcomes.csv`
- `postgres_prediction_events.csv`

Цель документа: зафиксировать проблемы качества прогноза и мониторинга факта рейса, чтобы не потерять контекст перед исправлением модели, monitor pipeline и разметки outcomes.

## Главный вывод

Плохое качество последних прогнозов связано не только с baseline-моделью. Есть как минимум три разных класса проблем:

1. Monitor неправильно интерпретирует перенос рейса на следующий день как успешное выполнение исходной даты.
2. Monitor слишком долго ждет финализации факта из-за `--finalize-lag-days=2`, хотя онлайн-табло airportus уже может показывать, что рейс текущего дня перенесен на завтра.
3. Predictor слишком оптимистично трактует найденное погодное окно и не учитывает расписание конкретных рейсов.

Из-за этого часть пользовательских промахов не попадает в метрики как ошибка, а часть реальных погодных рисков получает слишком слабый штраф.

## Проблема 1. Перенос на следующий день считается выполненным рейсом

Сейчас monitor строит outcome по множеству статусов за `flight_date`.

Код:

- `pipelines/evaluation/forecast_monitor.py`
- `choose_outcome_status(statuses)`

Текущая логика:

```text
если есть departed / arrived / in_flight -> completed
если есть cancelled / combined -> cancelled
если есть delayed -> needs_review
```

Проблема: логика не проверяет, что `actual_date` совпадает с `flight_date`.

Пример из архива:

- `flight_date=2026-06-06`
- рейсы `HZ-3034` и `HZ-3035`
- статус позже стал `departed/arrived`
- но `actual_date=2026-06-07`

Фактически для вопроса пользователя "полетит ли самолет 6 июня" ответ должен быть "нет, рейс перенесен на 7 июня". Но monitor засчитал `2026-06-06` как `completed`.

Последствие:

- пользователь видит промах прогноза `yes`;
- monitor может не считать это ошибкой;
- метрики качества становятся слишком оптимистичными;
- обучающая/оценочная разметка расходится с продуктовым смыслом прогноза.

Нужное поведение:

- если `actual_date > flight_date`, день исходного рейса должен быть `delayed_next_day`, `disrupted` или `cancelled_for_day`;
- для бинарной метрики "рейс будет в выбранный день" это должно считаться как `0`;
- отдельно можно хранить факт, что рейс физически выполнен позже.

## Проблема 2. Двухдневное окно финализации мешает оперативной оценке

В `pipelines/evaluation/forecast_monitor.py` сейчас есть параметр:

```text
--finalize-lag-days default=2
```

Финальность outcome считается примерно так:

```text
target_date <= today - finalize_lag_days
```

Это слишком грубо для текущего сервиса. Если официальный/рабочий источник табло уже показывает, что рейс перенесен на следующий день, monitor должен записывать outcome текущего дня сразу, без ожидания двух дней.

Практический пример:

- сайт табло: `https://airportus.ru/board/`
- по табло Южно-Сахалинска уже видно, что рейсы на Южно-Курильск текущего дня отложены на завтра;
- значит для выбранной даты уже понятно, что рейса "сегодня" не будет;
- monitor должен зафиксировать это как отмену/невыполнение дня и посчитать ошибку прогноза, если прогноз был `yes`.

Нужное поведение:

- не ждать `finalize_lag_days`, если есть сильный статус `combined`, `cancelled` или `delayed` с `actual_date > flight_date`;
- хранить `is_final=1` для таких случаев раньше;
- использовать `is_final=0` только для planned-only/scheduled/неполных наблюдений;
- возможно заменить один `status` на два поля:
  - `operational_status`: `completed`, `cancelled`, `delayed`, `combined`, `planned_only`, `needs_review`;
  - `same_day_outcome`: `completed_same_day`, `not_completed_same_day`, `unknown`.

## Проблема 3. Predictor не учитывает расписание рейсов

Сейчас погодное окно ищется в общем рабочем окне:

```text
08:00-20:00
```

Если найдено несколько часов подряд, predictor получает `flight_window_available=True` и сильный плюс к вероятности.

Проблема: окно может быть позже фактических рейсов дня.

Пример `2026-06-09`:

- рейсы по табло: около `10:20` и `13:45`;
- прогноз сервиса: `yes`;
- monitor/user evidence: задержка до `2026-06-10`;
- погодное окно у сервиса: примерно `14:00-20:00`;
- для рейсов `10:20` и `13:45` это окно уже позднее расписания.

Нужное поведение:

- учитывать расписание конкретных рейсов на дату из board collector;
- искать погодное окно вокруг реального расписания, например от `scheduled_departure - 1h` до `scheduled_return + 1h`;
- если расписания нет, использовать fallback `08:00-20:00`, но с меньшей уверенностью;
- не давать большой плюс за окно, которое начинается после основного вылета/возврата.

## Проблема 4. Низкая видимость без fog code слишком слабо влияет на решение

Код:

- `backend/app/services/weather.py`
- `_is_flight_opportunity_hour`
- `backend/app/services/predictor.py`
- `calculate_weather_adjustment`

Сейчас час исключается из летного окна по видимости только если:

```text
visibility < WEATHER_FLIGHT_WINDOW_MIN_VISIBILITY
и есть fog weather_code
```

Если видимость плохая, но Open-Meteo не поставил weather code `45/48`, час может остаться пригодным.

В predictor штрафы тоже мягкие:

- `visibility <= 1000` без fog code: около `-0.03`;
- `fog_low_cloud_risk_level=high` без severe condition: около `-0.03`;
- `flight_window_available=True`: `+0.12`.

Итог: найденное окно может перекрыть плохую видимость.

Пример `2026-06-09`:

- сервис видел `visibility=390`;
- `weather_code=2`;
- fog code отсутствовал;
- `flight_window_available=True`;
- прогноз стал `yes`.

Нужное поведение:

- низкая видимость должна быть самостоятельным сильным blocker-сигналом, особенно для `<=1000 м`;
- не полагаться только на Open-Meteo fog code;
- отдельно проверить edge case `2026-06-03`, где видимость была очень плохой, но рейс все равно выполнен. Простое ужесточение порога может ухудшить false negative по таким дням.

## Проблема 5. Историческая база тянет прогноз к "yes"

Для последних дат базовая вероятность часто около `0.56-0.58`.

При пороге `0.55` для близких дат даже небольшой положительный weather adjustment делает прогноз `yes`.

Пример:

- `2026-06-08`: base около `0.568`, найденное окно дало плюс, итог `0.66`, decision `yes`, факт не выполнен в этот день.
- `2026-06-09`: base около `0.580`, найденное окно дало плюс, итог `0.65`, decision `yes`, табло показывает перенос на следующий день.

Нужное поведение:

- после исправления same-day outcomes пересчитать качество;
- проверить, не слишком низкий порог `0.55` для ближних дат в плохую погоду;
- проверить, не слишком сильный бонус `+0.12` за летное окно;
- возможно перейти от эвристики к модели, обученной на `same_day_outcome`.

## Проверенные даты

### 2026-06-03

- Факт по архиву: `completed`.
- Сырые статусы: `departed`, `arrived`, `in_flight`, `scheduled`.
- Сервис в день прогноза давал `no`.
- Это реальный false negative.
- Погода была плохая по видимости и низкой облачности, но рейс состоялся.
- Этот день важен как counterexample: нельзя просто сделать правило "плохая видимость -> всегда no".

### 2026-06-04

- Факт по архиву: `completed`.
- Ближний прогноз стал `yes`.
- Более ранние прогнозы за несколько дней могли быть `no`.
- Ошибка скорее связана с нестабильностью прогноза погоды по горизонту.

### 2026-06-05

- По архиву рейсы выполнены в тот же день.
- Есть задержки, но затем `departed/arrived` с `actual_date=2026-06-05`.
- По текущим данным это не ошибка прогноза `yes`.

### 2026-06-06

- Сервис давал `yes`.
- Monitor засчитал `completed`.
- Но сырое табло показывает перенос на `2026-06-07`.
- Для same-day-продукта это должен быть промах прогноза.

### 2026-06-08

- Сервис давал `yes`.
- Табло показывает задержку/совмещение на следующие даты.
- В monitor стоит `cancelled`, но `is_final=0` из-за finalize lag.
- Это должно становиться финальным outcome раньше, как только табло показывает перенос/совмещение.

### 2026-06-09

- Сервис давал `yes`.
- Табло показывает задержку до `2026-06-10`.
- В monitor пока `needs_review`.
- Для продукта "будет ли рейс сегодня" это уже должно считаться невыполнением дня.

## Предлагаемый план исправления

1. Ввести явную same-day outcome модель.
2. В monitor учитывать `actual_date` относительно `flight_date`.
3. Убрать жесткое ожидание `finalize_lag_days=2` для сильных статусов:
   - `cancelled`
   - `combined`
   - `delayed` с `actual_date > flight_date`
   - `departed/arrived` с `actual_date > flight_date`
4. Добавить regression tests на даты:
   - `2026-06-03`: плохая погода, но completed same-day;
   - `2026-06-06`: departed/arrived next-day, но not completed same-day;
   - `2026-06-08`: combined next-day, not completed same-day;
   - `2026-06-09`: delayed to next day, not completed same-day.
5. Подключить расписание рейсов к predictor weather window.
6. Пересчитать метрики после исправления labels.
7. Только после этого менять веса weather adjustment и decision threshold.

## Статус исправления 2026-06-09

Реализованы первые guardrails для проблем 1-5:

- Monitor теперь считает перенос на следующую дату как невыполнение исходного дня.
- Monitor не ждет двухдневный lag, если табло уже показывает сильный финальный сигнал: перенос, совмещение, отмену или выполнение на следующую дату.
- Predictor читает schedule snapshot из `data/raw/flight_status/kunashir_flight_status_hourly.csv`.
- Если табло уже показывает перенос рейса на следующую дату, вероятность capped low и решение становится `no`.
- Если погодное окно найдено слишком поздно относительно первого вылета по табло, оно не дает оптимистичный бонус.
- Низкая видимость теперь сама исключает час из летного окна, без обязательного fog weather code.
- Ближний порог для `yes` на горизонте `0-2` дня стал более консервативным.

Важное ограничение для будущих backtest: schedule-aware predictor нельзя честно прогонять по старым датам на полном текущем `kunashir_flight_status_hourly.csv`, потому что в нем уже есть будущие относительно прогноза факты табло. Для честного исторического backtest нужно использовать только строки табло, которые существовали на момент `prediction_created_at`.

## Важный продуктовый выбор

Нужно явно определить, что предсказывает сервис:

- "самолет физически когда-нибудь выполнит этот рейс" или
- "рейс будет выполнен в выбранную дату".

Текущий пользовательский смысл ближе ко второму. Поэтому перенос на завтра должен считаться `no` для исходной даты, даже если самолет в итоге улетел следующим днем.
